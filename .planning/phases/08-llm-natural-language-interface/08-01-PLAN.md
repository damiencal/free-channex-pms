  **SYSTEM_PROMPT** — A string constant containing:
  - Role: "You are a PostgreSQL SQL generator for a vacation rental management system."
  - TASK: "Convert the user's natural language question into a single valid PostgreSQL SELECT query."
  - RULES section:
    - Generate ONLY SELECT. Never INSERT, UPDATE, DELETE, DROP, CREATE, ALTER.
    - Never perform arithmetic. SQL aggregations (SUM, AVG, COUNT) compute all numbers.
    - If the question is ambiguous (missing property name, time period), ask ONE clarifying question instead of guessing. Do NOT output SQL when clarifying.
    - If the question is outside financial/booking scope, respond: "I can answer questions about revenue, expenses, bookings, and occupancy."
    - Always output SQL in a ```sql ... ``` code fence. Nothing else outside the fence.
  - DATABASE SCHEMA section with these 6 tables (use the exact column types from the ORM models):
    - properties (id INTEGER PK, slug VARCHAR(64), display_name VARCHAR(255))
    - bookings (id INTEGER PK, platform VARCHAR(32), guest_name VARCHAR(255), check_in_date DATE, check_out_date DATE, net_amount NUMERIC(10,2), property_id INTEGER FK)
    - journal_entries (id INTEGER PK, entry_date DATE, description VARCHAR(512), source_type VARCHAR(64), property_id INTEGER FK nullable)
    - journal_lines (id INTEGER PK, entry_id INTEGER FK, account_id INTEGER FK, amount NUMERIC(12,2) — positive=debit, negative=credit)
    - accounts (id INTEGER PK, number INTEGER, name VARCHAR(128), account_type VARCHAR(32) — 'asset','liability','equity','revenue','expense')
    - expenses (id INTEGER PK, expense_date DATE, amount NUMERIC(12,2) — always positive, category VARCHAR(64), description VARCHAR(512), vendor VARCHAR(255), attribution VARCHAR(32), property_id INTEGER FK)
  - SIGN CONVENTION note: Revenue journal_lines have NEGATIVE amounts (credits). To get positive revenue: use `-SUM(journal_lines.amount)` or `SUM(journal_lines.amount) * -1` for revenue accounts (account_type = 'revenue').
  - PROPERTY NAMES note: Properties are identified by slug ('jay', 'minnie') or display_name ('Jay', 'Minnie'). Use case-insensitive ILIKE for name matching.
  - EXAMPLE QUERIES section with 2-3 examples:
    - "How much did Jay make this month?" -> SELECT with SUM on journal_lines joined to revenue accounts, filtered by property slug and current month
    - "Show all Airbnb bookings in January 2026" -> SELECT from bookings WHERE platform = 'airbnb' AND date range

  **build_sql_messages(user_question: str, history: list[dict]) -> list[dict]:**
  - Returns message list for Ollama chat API
  - First message: {"role": "system", "content": SYSTEM_PROMPT}
  - Then: last 10 items from history (already formatted as {"role": "user"|"assistant", "content": str})
  - Finally: {"role": "user", "content": user_question}

  **NARRATIVE_SYSTEM_PROMPT** — A string constant for Phase B:
  - Role: "You describe SQL query results in plain, conversational English for a non-technical user."
  - Rules:
    - Use the provided SQL results to write your answer. Never compute or estimate numbers.
    - Format dollar amounts as $X,XXX.XX.
    - Be concise — 1-3 sentences.
    - If results are empty, say so clearly and suggest an alternative query.

  **build_narrative_messages(user_question: str, sql: str, rows: list[dict], columns: list[str]) -> list[dict]:**
  - System message: NARRATIVE_SYSTEM_PROMPT
  - User message containing:
    - "The user asked: {user_question}"
    - "This SQL was executed: {sql}"
    - "Results ({len(rows)} rows):" followed by a formatted text table of the first 50 rows (truncate if more, note "showing first 50 of N rows")
    - If rows is empty: "The query returned no results."

  **extract_sql_from_response(text: str) -> str:**
  - Try regex: ```(?:sql)?\s*(.*?)``` with DOTALL|IGNORECASE
  - Fallback: look for `SELECT` keyword to end of text
  - If neither found, return the original text (let the caller decide — if it contains no SQL, it's a clarification)
  - Strip whitespace from result
  - Do NOT raise on no-SQL — the absence of SQL means the LLM is asking a clarification question or declining to answer

  Important: Do NOT import from app.models — the schema is hardcoded in the prompt string to keep it minimal and controlled. The prompt should NOT dynamically read the database schema.

  <files>app/query/sql_validator.py, app/query/ollama_client.py</files>
  <action>
  1. Create `app/query/sql_validator.py`:

  **SQLValidationError(Exception):** Custom exception for validation failures.

  **validate_sql(sql: str) -> str:**
  - Parse with `sqlglot.parse(sql.strip(), read="postgres")`
  - Catch parse exceptions -> raise SQLValidationError(f"SQL parse error: {e}")
  - Check `not statements or statements[0] is None` -> raise SQLValidationError("Empty SQL")
  - Check `len(statements) > 1` -> raise SQLValidationError("Multiple statements not allowed")
  - Check `not isinstance(statements[0], exp.Select)` -> raise SQLValidationError(f"Only SELECT is allowed. Got: {type(statements[0]).__name__}")
  - Return `sql.strip()` on success

  Import: `import sqlglot` and `from sqlglot import expressions as exp`

  2. Create `app/query/ollama_client.py`:

  **get_ollama_client() -> AsyncClient:**
  - Module-level `_client: AsyncClient | None = None`
  - On first call: create `AsyncClient(host=get_config().ollama_url)` and cache in `_client`
  - Return cached client on subsequent calls
  - Import `from ollama import AsyncClient` and `from app.config import get_config`

  Note: Do NOT use `from app.config import get_config` at module level for the actual call — call `get_config()` inside the function body so it works even if config isn't loaded at import time.
  </action>
