  <action>
Create CONTRIBUTING.md at repo root covering development setup and contribution guidelines.

**Structure:**

1. **Introduction**: Brief welcome. State the project uses Apache 2.0 license. Contributions welcome.

2. **Development Setup**: Step-by-step local dev environment setup (bare metal / dev mode):
   - Prerequisites: Python 3.12+, Node.js 18+, PostgreSQL 16, uv (Python package manager)
   - Clone the repo
   - Backend setup:
     - `uv sync` to install Python dependencies
     - Copy `.env.example` to `.env` and configure DATABASE_URL pointing to local Postgres
     - Copy config examples to `config/base.yaml` and `config/{slug}.yaml`
     - `alembic upgrade head` to run migrations
     - `uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload` to start dev server
   - Frontend setup:
     - `cd frontend && npm install`
     - `npm run dev` (Vite dev server with HMR)
     - Frontend proxies API calls to `http://localhost:8000`
   - Docker setup (alternative):
     - `docker compose up` handles everything
     - For development with hot-reload, mount source as volume (provide example override)

3. **Project Structure**: Brief overview of directory layout:
   - `app/` — Python backend (FastAPI)
   - `app/api/` — API route modules
   - `app/models/` — SQLAlchemy models
   - `app/ingestion/` — CSV parsing adapters
   - `app/accounting/` — Revenue recognition, journal entry logic
   - `app/compliance/` — Resort form filling and submission
   - `app/communication/` — Guest messaging orchestration
   - `frontend/` — React SPA (Vite + TypeScript)
   - `config/` — YAML property configs (gitignored except examples)
   - `templates/` — Jinja2 message templates
   - `pdf_mappings/` — PDF form field mappings
   - `alembic/` — Database migrations

4. **Code Style**:
   - Python: Follow existing patterns. Type hints on all function signatures. Pydantic models for request/response. structlog for logging.
   - TypeScript: Follow existing patterns. Functional components with hooks. shadcn/ui for UI components.
   - No linter/formatter configured yet — match existing style.

5. **Making Changes**:
   - Fork the repo
   - Create a feature branch from `main`
   - Make changes with clear commit messages
   - Test your changes locally (both backend startup and frontend build)
   - Submit a PR against `main`

6. **Reporting Issues**:
   - Use GitHub Issues
   - Include: steps to reproduce, expected behavior, actual behavior, environment details
   - For feature requests: describe the use case, not just the solution

7. **License**: Contributions are licensed under Apache 2.0.

**Tone**: Welcoming but concise. Do NOT use emojis.

  <files>docs/deployment.md</files>
  <action>
Create `docs/` directory and `docs/deployment.md` — a step-by-step self-hosting guide.

**CRITICAL PITFALLS TO AVOID (from RESEARCH.md):**
- SMTP must be presented as REQUIRED for compliance/communication features (Pitfall 5)
- Ollama must be presented as OPTIONAL with graceful degradation (Pitfall 6)
- Distinguish localhost (verification) from production URLs (Pitfall 1)
- Config fields must match app/config.py exactly — read AppConfig and PropertyConfig (Pitfall 2)

**Structure:**

1. **Title**: "Deployment Guide"

2. **Prerequisites**:
   - Docker and Docker Compose (required)
   - Ollama (optional — natural language queries disabled without it)
   - SMTP-capable email account (required for resort form submission and operator notifications)

3. **Step 1: Clone the Repository**:
   ```bash
   git clone https://github.com/captainarcher/roost.git
   cd roost
   ```

4. **Step 2: Configure Environment Variables**:
   - `cp .env.example .env`
   - Explain each variable in `.env`:
     - `POSTGRES_DB` — database name (default: `rental_management`)
     - `POSTGRES_USER` — database user (default: `rental`)
     - `POSTGRES_PASSWORD` — **change this** to a secure password
     - `DATABASE_URL` — must match the user/password/host above (host is `roost-db` inside Docker)
     - `OLLAMA_URL` — Ollama API URL (default: `http://host.docker.internal:11434`). Remove or leave if not using Ollama.
     - `SMTP_HOST` — SMTP server (e.g., `smtp.gmail.com`)
     - `SMTP_PORT` — SMTP port (e.g., `587` for STARTTLS)
     - `SMTP_USER` — email address for SMTP auth
     - `SMTP_PASSWORD` — app password (not regular password — for Gmail, create an App Password)
     - `SMTP_FROM_EMAIL` — sender email address

5. **Step 3: Configure Base Settings**:
   - `cp config/base.example.yaml config/base.yaml`
   - Walk through each field in base.yaml (read from config/base.example.yaml):
     - `ollama_url`, `ollama_model` — LLM settings (optional)
     - `archive_dir` — where processed CSVs are archived
     - `airbnb_fee_model`, `airbnb_host_fee_rate` — Airbnb fee structure
     - `confirmations_dir` — where booking confirmation PDFs are stored
     - `pdf_template_path`, `pdf_mapping_path` — resort PDF form configuration
     - `auto_submit_threshold` — days before check-in to auto-submit resort form (set to 0 to disable auto-submit)
     - `resort_contact_name` — resort contact name for form submission email

6. **Step 4: Configure Properties**:
   - `cp config/config.example.yaml config/{your-property-slug}.yaml`
   - Walk through EVERY field in config.example.yaml (read from the file):
     - Required: slug, display_name, lock_code, site_number, resort_contact_email, resort_checkin_instructions, host_name, host_phone, listing_slug_map
     - Guest communication: wifi_password, address, check_in_time, check_out_time, parking_instructions, local_tips
     - Custom fields: Arbitrary key-value pairs accessible in templates as `{{ custom.key }}`
   - Explain listing_slug_map: maps platform listing titles/IDs to the property slug for automatic booking-to-property assignment
   - Explain that ONE .yaml file per property — add as many properties as needed

7. **Step 5: Start Roost**:
   ```bash
   docker compose up -d
   ```
   - First startup: Alembic migrations run automatically, chart of accounts seeded
   - Database initialization takes ~10 seconds
   - Verify startup:
     ```bash
     docker compose logs roost-api --tail 50
     ```
   - Look for: "Roost ready" log line (or similar from app/main.py)

8. **Step 6: Verify Installation**:
   ```bash
   curl http://localhost:8000/health
   ```
   - Explain the response fields (database, ollama, properties)
   - If database shows "connected" and properties appear, installation is successful
   - Access the dashboard at `http://localhost:8000`

9. **Step 7: Import Your First Data**:
   - Brief walkthrough of uploading an Airbnb CSV via curl or the dashboard UI
   - Link to API docs for more import options

10. **Ollama Setup** (separate section — optional):
    - Install Ollama on the host machine
    - Pull the default model: `ollama pull llama3.2:latest`
    - Ensure Ollama is accessible from Docker (host.docker.internal)
    - Verify: health endpoint should show `"ollama": "available"`
    - If not using Ollama: natural language query feature is disabled, all other features work normally

11. **Message Templates** (separate section):
    - Default templates in `templates/default/`
    - Override per-property by creating `templates/{slug}/` with same filenames
    - Templates use Jinja2 syntax with property config fields as variables

12. **Resort PDF Forms** (separate section):
    - PDF form template in `pdf_mappings/`
    - JSON mapping file controls which form fields get which data
    - Customizable for different resort form formats

13. **Updating**:
    ```bash
    git pull
    docker compose up -d --build
    ```
    - Alembic migrations run automatically on startup

14. **Troubleshooting**:
    - Common issues: wrong POSTGRES_PASSWORD in DATABASE_URL, SMTP auth failure, Ollama not reachable
    - Logs: `docker compose logs roost-api`
    - Config validation: app fails fast on startup with clear error messages listing missing fields

**Tone**: Copy-paste friendly. Every command is in a code block. Assumes Docker knowledge but not FastAPI/Postgres experience. Do NOT use emojis.
