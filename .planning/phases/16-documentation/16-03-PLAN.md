**Structure:**

1. **Title**: "Architecture Overview"

2. **Overview**: 2-3 paragraph summary. Roost is a self-hosted vacation rental operations platform built with FastAPI, PostgreSQL, and React. It automates end-to-end: booking ingestion from multiple platforms, double-entry accounting, resort compliance form submission, and guest communication — all triggered by a single CSV upload.

3. **System Components** (with Mermaid diagram #1 — `graph TD` flowchart):
   Create a system component diagram showing:
   - Frontend (React SPA) → FastAPI Backend
   - FastAPI Backend → PostgreSQL (via SQLAlchemy)
   - FastAPI Backend → Ollama (optional, for NL queries)
   - FastAPI Backend → SMTP (for resort form email)
   - Backend subsystems: Ingestion, Accounting, Compliance, Communication, Query, Dashboard, Reports

   Then describe each subsystem:
   - **Ingestion** (`app/ingestion/`): CSV parsing via Polars adapters (Airbnb, VRBO, Mercury), RVshare manual JSON entry. Idempotent — re-imports are safe. Triggers background tasks.
   - **Accounting** (`app/accounting/`): Double-entry bookkeeping. Chart of accounts (seeded by Alembic). Revenue recognition creates balanced journal entries. Expense tracking, loan management, bank reconciliation.
   - **Compliance** (`app/compliance/`): PDF form filling with pypdf. Auto-submits resort booking form via email when check-in is within threshold. Preview mode for operator approval. Urgency tracking via APScheduler daily cron.
   - **Communication** (`app/communication/`): Platform-aware guest messaging. Airbnb: native (platform handles delivery, Roost logs `native_configured`). VRBO/RVshare: Roost generates message, emails operator, operator sends manually. Pre-arrival messages scheduled via APScheduler.
   - **Query** (`app/api/query.py`): Two-phase LLM pipeline. Phase A: text-to-SQL (non-streaming, temperature 0.1, sqlglot validation). Phase B: narrative generation (streaming SSE, temperature 0.3). All numbers from SQL, never LLM arithmetic.
   - **Dashboard** (`app/api/dashboard.py`): Aggregated metrics, booking calendar, occupancy rates, pending action items.
   - **Reports** (`app/api/reports.py`): P&L, balance sheet, income statement with flexible period filtering.

4. **The Automation Pipeline** (with Mermaid diagram #2 — `sequenceDiagram`):
   This is the core differentiator. Create a sequence diagram showing:
   - User uploads CSV → Ingestion endpoint → Returns response
   - Background: Booking inserted → Revenue Recognition (journal entries created)
   - Background: Booking inserted → Resort Form Submission (if within threshold)
   - Background: Booking inserted → Welcome Message scheduled
   - APScheduler: Pre-arrival message fires 2-3 days before check-in

   Write a detailed narrative explaining each step:
   - CSV upload is synchronous — response returns immediately with import summary
   - Background tasks fire asynchronously after response (FastAPI BackgroundTasks)
   - Revenue recognition creates debit/credit journal entries automatically
   - Resort form submission: pypdf fills the PDF, attaches booking confirmation, emails resort contact
   - Welcome message: for Airbnb, logged as native_configured; for VRBO/RVshare, operator notified via email
   - Pre-arrival message: scheduled job sends 2-3 days before check-in with property details and lock code

   3. Verify database connection — FAIL-FAST
   3b. Sync properties from YAML → database (upsert)
   4. Check Ollama connectivity — NON-FATAL
   5. Start APScheduler (compliance urgency check daily at 08:00)
   6. Rebuild pre-arrival message scheduler jobs from database

   Explain the fail-fast philosophy: if config is wrong, the app refuses to start with a clear error message. This prevents runtime surprises.

7. **Key Design Decisions**: Table or list covering:

   - Double-entry accounting: why (audit trail, balanced ledger)
   - Text-to-SQL (not LLM arithmetic): why (accuracy, verifiability)
