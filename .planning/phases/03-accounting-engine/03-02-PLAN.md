<objective>
Implement revenue recognition: when a booking is processed (via operator-triggered API call), create the appropriate double-entry journal entries. Airbnb uses a deferred-then-recognized pattern (Unearned Revenue on booking creation, Rental Income on payout). VRBO and RVshare use single-event recognition. Adjustment rows (credits, clawbacks, corrections) each produce their own journal entry. Fee model is configurable via AppConfig.

Purpose: This is ACCT-01 + ACCT-05 + ACCT-09 combined -- the core accounting logic that turns bookings into ledger entries. Revenue recognition is intentionally operator-triggered (not automatic on import) because this is a batch-oriented tool where the operator reviews imports before posting to the ledger.

Output: `app/accounting/revenue.py` with functions for all three platforms plus adjustment handling, and config additions for fee model.
</objective>

<task type="auto">
  <name>Task 3: Implement revenue recognition module with adjustment and discount handling</name>
  <files>app/accounting/revenue.py</files>
  <action>
Create `app/accounting/revenue.py` with the following functions:

**Helper: `_get_account_by_name(db: Session, name: str) -> Account`**
- Query Account by name, raise ValueError if not found
- Cache results in a module-level dict (only ~20 accounts; avoids repeated SELECTs)

**Helper: `_calculate_platform_fee(net_amount: Decimal, fee_rate: Decimal) -> tuple[Decimal, Decimal]`**
- For host-only model: gross = net / (1 - fee_rate), fee = gross - net
- For split-fee model: fee = net * fee_rate / (1 - fee_rate), gross = net + fee (the net amount in the CSV is after the 3% deduction)
- Use `Decimal.quantize(Decimal("0.01"), rounding=ROUND_HALF_UP)` for all results
- Returns (gross_amount, fee_amount)

**`recognize_booking_revenue(db: Session, booking: Booking, config: AppConfig) -> list[JournalEntry | None]`**
NOTE: Returns a LIST of journal entries (not a single entry) because a booking may produce multiple entries (payout + adjustment entries). The caller (API endpoint) triggers this explicitly -- it is NOT called automatically on import.

- For Airbnb bookings: inspect `booking.raw_platform_data["rows"]` to determine event types
  - If rows contain a "Payout" type row: this is a completed payout. Create full revenue recognition entry:
    - Dr. Mercury Checking (+net_amount)
    - Cr. Rental Income (-gross_amount)
    - Dr. Platform Fees (+fee_amount)
  - Also check if an Unearned Revenue entry exists for this booking (source_id pattern `booking_unearned:airbnb:{platform_booking_id}`). If it does, create a clearing entry to zero out the unearned revenue.
  - source_id: `booking_payout:airbnb:{booking.platform_booking_id}`
  - entry_date: booking.check_in_date (per CONTEXT.md: revenue at payout, which is approximately check-in)
  - After processing the Payout row, call `record_adjustment_entries(db, booking)` to handle any adjustment rows (see below). Append results to the return list.
- For VRBO bookings: single recognition entry (same Dr/Cr structure as Airbnb payout, but no deferred step). Fee rate: use a reasonable default (VRBO takes ~3% per RESEARCH.md, but since VRBO net_amount in CSV is already net, treat the full amount as net and skip fee reconstruction unless fee data is available). For VRBO/RVshare, if fee data is not available in raw_platform_data, record net_amount as gross (no fee line). source_id: `booking_payout:vrbo:{booking.platform_booking_id}`
- For RVshare bookings: same as VRBO -- single entry, net=gross if no fee data. source_id: `booking_payout:rvshare:{booking.platform_booking_id}`
- Pass `property_id=booking.property_id` to create_journal_entry
- Returns list of JournalEntry results (some may be None if idempotent skip)

**`record_adjustment_entries(db: Session, booking: Booking) -> list[JournalEntry | None]`**
Per CONTEXT.md locked decision: "each adjustment row (credit, clawback, correction) produces its own journal entry against the original booking."

- Define ADJUSTMENT_ROW_TYPES = {"Adjustment", "Credit", "Resolution"} -- these are the Airbnb CSV row types that represent post-booking adjustments
- Iterate `booking.raw_platform_data["rows"]` and filter for rows whose "type" field is in ADJUSTMENT_ROW_TYPES
- For each adjustment row:
  - Extract the row's amount (may be positive or negative)
  - If amount > 0 (credit to host): Dr. Mercury Checking (+amount), Cr. Rental Income (-amount)
  - If amount < 0 (clawback from host): Dr. Rental Income (+abs(amount)), Cr. Mercury Checking (-abs(amount))
  - source_type: "adjustment"
  - source_id: `adjustment:{booking.platform}:{booking.platform_booking_id}:{row_index}` (row_index to ensure uniqueness per adjustment row)
  - entry_date: date from the adjustment row if available, otherwise booking.check_in_date
  - description: f"Adjustment ({row['type']}): {booking.platform_booking_id}" with row details
  - property_id: booking.property_id
- Return list of JournalEntry results (idempotent via source_id)

**Discount handling (deferred -- LOW confidence on CSV availability):**
- Within `record_adjustment_entries`, also check for rows with type "Discount" or "Promotional discount" (case-insensitive check)
- If a discount row is found:
  - Dr. Promotional Discounts (account 4010) (+abs(amount)) -- contra-revenue debit
  - Cr. Rental Income (account 4000) (-abs(amount)) -- reduces recognized revenue
  - source_id: `discount:{booking.platform}:{booking.platform_booking_id}:{row_index}`
  - source_type: "adjustment"
- If NO discount rows are found in raw_platform_data, log a DEBUG-level message: `"No discount rows found for booking {platform_booking_id} -- discount data may not be available in CSV imports"` using Python logging. Do NOT error or warn -- this is expected given LOW confidence from research that discount rows appear in Airbnb CSVs.

**`create_unearned_revenue_entry(db: Session, booking: Booking) -> JournalEntry | None`**
- For Airbnb bookings that have a Reservation row but no Payout row yet
- Dr. Accounts Receivable (+net_amount)
- Cr. Unearned Revenue (-net_amount)
- source_id: `booking_unearned:airbnb:{booking.platform_booking_id}`
- entry_date: the date from the Reservation row (or booking.check_in_date as fallback)
- property_id: booking.property_id

**`reverse_journal_entry(db: Session, original_source_id: str, reversal_date: date, reason: str) -> JournalEntry | None`**
- Find original JournalEntry by source_id
- Create reversal entry with negated line amounts
- source_id: `reversal:{original_source_id}`
- source_type: "reversal"
- description: f"Reversal of: {original.description} -- {reason}"
- Both original and reversal remain in DB for audit trail

IMPORTANT:
- All monetary arithmetic uses Decimal, never float
- Convert config.airbnb_host_fee_rate to Decimal before arithmetic: `Decimal(str(config.airbnb_host_fee_rate))`
- Use `from decimal import Decimal, ROUND_HALF_UP` for rounding
- Every journal entry must balance (create_journal_entry enforces this, but verify your line construction is correct)
- Idempotency is handled by source_id uniqueness in create_journal_entry
- Use `import logging` and `logger = logging.getLogger(__name__)` for the discount debug log
