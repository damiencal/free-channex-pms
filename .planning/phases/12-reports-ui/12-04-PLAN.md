  <action>
OVERWRITE the stub `frontend/src/components/reports/IncomeStatementTab.tsx` with the full Income Statement viewer.

**Imports:**
- `useState` from React
- `useIncomeStatement` from `@/hooks/useReports`
- `ReportFilters` from `./ReportFilters`
- `ReportSection` from `./ReportSection`
- `MonthlyTable` from `./MonthlyTable`
- `EmptyState` from `@/components/shared/EmptyState`
- `ErrorAlert` from `@/components/shared/ErrorAlert`
- `Skeleton` from `@/components/ui/skeleton`
- Types: `IncomeStatementParams`, `IncomeStatementTotalsResponse`, `IncomeStatementMonthlyResponse` from `@/api/reports`

**Number formatting** (same utility as BalanceSheet/PLTab — define locally):
- `formatAmount(value: string)` → { display, isNegative, isZero }
- `AmountCell` local component

**Sub-view state:**
- `useState<'totals' | 'monthly'>('totals')`
- Two small buttons: "Totals" | "Monthly" — same toggle pattern as PLTab

**Key difference from P&L:** Income Statement uses `breakdown` parameter to switch between totals and monthly AT THE API LEVEL. When user selects "Totals", send `breakdown: 'totals'`. When user selects "Monthly", send `breakdown: 'monthly'`. This means the `generate()` call must include the breakdown value.

**Implementation approach:**
- The active sub-view determines the `breakdown` value sent to the API
- When the user switches sub-view AND has already generated a report, automatically re-generate with the new breakdown. Use a `useEffect` that watches the sub-view and re-calls `generate()` if `hasGenerated` is true.
- OR: Keep it simpler — require the user to click Generate again after switching. Show a message "Click Generate to load the {totals/monthly} view." This avoids auto-fetch complexity. **Go with this simpler approach** — it matches the "Generate button required" CONTEXT decision.

**Component structure:**

1. Call `useIncomeStatement()` to get `{ data, isFetching, isError, error, refetch, generate, hasGenerated }`

2. `activeView: 'totals' | 'monthly'` state

3. Render `ReportFilters` with `mode="range"`, `onGenerate` handler that calls `generate({ start_date, end_date, breakdown: activeView })`. When the user clicks Generate, it uses the currently active sub-view as the breakdown.

4. Render sub-view toggle buttons below filters (only visible if hasGenerated): "Totals" | "Monthly"

5. **State rendering logic** (same pattern):
   - `!hasGenerated`: prompt "Select a date range and click Generate to view the income statement."
   - `isFetching` without data: skeleton blocks
   - `isError`: ErrorAlert with retry
   - Empty data check

6. **Determine response type:** Check `data.breakdown` to determine which shape we have:
   - If `data.breakdown === 'totals'`: render totals view
   - If `data.breakdown === 'monthly'`: render monthly view
   - If the activeView doesn't match data.breakdown (user switched view but hasn't re-generated): show a message "Click Generate to load the {activeView} view."

7. **TOTALS view** (when `data.breakdown === 'totals'`, cast as `IncomeStatementTotalsResponse`):

   **Revenue ReportSection** (`title="Revenue"`, `total` formatted):
   - Table with 2 columns: Account | Amount
   - For each entry in `data.revenue.by_account`: show account name and formatted amount
   - Subtotal row: "Total Revenue" | data.revenue.total

   **Expenses ReportSection** (`title="Expenses"`, `total` formatted):
   - Table with 2 columns: Account | Amount
   - For each entry in `data.expenses.by_account`: show account name and formatted amount
   - Subtotal row: "Total Expenses" | data.expenses.total

   **Net Income grand total row:**
   - `bg-muted/60 font-semibold border-t-2`
   - "Net Income" | formatted data.net_income

8. **MONTHLY view** (when `data.breakdown === 'monthly'`, cast as `IncomeStatementMonthlyResponse`):

   **Account union computation** (per 12-RESEARCH.md Open Question 3):
   - Collect ALL unique revenue account names across ALL months: `data.months.flatMap(m => Object.keys(m.revenue.by_account))`, deduplicate
   - Collect ALL unique expense account names across ALL months: same approach with `m.expenses.by_account`
   - Collect ordered month keys: `data.months.map(m => m.month)`, already sorted

   **Revenue MonthlyTable:**
   - For each unique revenue account: create a row with `label = accountName`, `values = { [month]: data.months.find(m => m.month === month)?.revenue.by_account[accountName] ?? '0' }` for each month, `total = data.totals.revenue.by_account[accountName] ?? '0'`
   - If an account doesn't appear in a month's by_account, the value is missing — MonthlyTable handles this by showing "—"
   - Add subtotal row: label = "Total Revenue", values = each month's revenue.total, total = data.totals.revenue.total, isSubtotal = true

   **Expenses MonthlyTable:**
   - Same approach for expense accounts
   - Add subtotal row for Total Expenses

   **Net Income row:**
   - Can be shown as a single grand total row in a separate small MonthlyTable, OR as a standalone styled div below the expenses section
   - **Go with:** A Net Income MonthlyTable row with `isGrandTotal: true`, values = each month's net_income, total = data.totals.net_income

   Wrap each MonthlyTable in a ReportSection:
   - Revenue ReportSection with MonthlyTable inside
   - Expenses ReportSection with MonthlyTable inside
   - Net Income as a styled row outside sections (same as P&L and Balance Sheet pattern)
