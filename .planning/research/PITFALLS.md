# Pitfalls Research

**Domain:** Self-hosted vacation rental management suite (Airbnb, VRBO, RVshare, Mercury bank)
**Researched:** 2026-02-26
**Confidence:** MEDIUM — verified against official docs and community sources; RVshare API specifics LOW confidence due to sparse official documentation

---

## Critical Pitfalls

### Pitfall 1: Airbnb iCal Export Strips All Useful Guest Data

**What goes wrong:**
Developers plan to parse Airbnb iCal exports to get guest name, confirmation code, phone number, and email. Since December 2019, Airbnb removed all of this from iCal. The exported feed now contains only "Reserved" as the event title, check-in/check-out dates, the last 4 digits of the guest's phone number, and a link back to the Airbnb reservation page. There is no guest name, no confirmation code, no email. Automated messaging and PDF form filling cannot be triggered from iCal alone.

**Why it happens:**
Developers discover that Airbnb offers a calendar export URL and assume it contains booking details. Early tutorials and Stack Overflow answers still reference the pre-2019 format that included guest data. The limitation is not prominently documented.

**How to avoid:**
Do not use iCal as the primary data source for booking details. Use CSV transaction/earnings exports from the Airbnb earnings dashboard as the financial data source. For guest details (name, email, phone), screen-scrape the Airbnb reservation page (fragile, ToS risk) or parse the platform's booking confirmation email sent to the host's inbox. Email parsing is the most practical non-API approach: Airbnb sends a structured notification email for each booking that contains guest name, dates, confirmation code, and payout amount.

**Warning signs:**
- Parsing iCal events and getting only "Reserved" strings with no guest fields
- Guest name showing as empty in the data pipeline
- Confirmation codes missing from processed bookings

**Phase to address:**
Platform data ingestion phase — design the ingestion pipeline around email parsing or CSV exports from day one, not iCal. Document clearly in the architecture decision record.

### Pitfall 2: Export CSV Format Is Silently Brittle

**What goes wrong:**
Airbnb, VRBO, and Mercury each provide downloadable CSV exports. These formats are not versioned APIs — they are UI-generated files that change without announcement. Known Airbnb CSV issues reported by users: apostrophes prepended to numeric fields preventing summation, dates in mixed US/UK formats within the same file, some dates stored as text rather than date types, missing quote marks around fields that contain commas causing multi-column spillage. A parser that works today may silently produce corrupted financial data after a platform UI update.

**Why it happens:**
Export CSV files are generated by frontend teams for human consumption in Excel, not for machine ingestion. They are treated as a presentation layer, not an API contract. Platforms do not publish changelogs for CSV format changes.

**How to avoid:**
- Add a schema validation step as the first thing the ingestion pipeline does after reading a CSV. Validate that expected column headers are present and that numeric fields actually parse as numbers.
- Store raw source files (unmodified CSVs) alongside processed records so you can re-parse historical data when the parser is updated.
- Write parsers to be column-name-based (not index-based) — header name changes are detectable; column order shifts are not.
- Include a hash of the first-row header line in logs; alert if it changes.
- For Airbnb, prefer the Transaction History CSV from the earnings dashboard over the calendar export; it contains payout amounts, fees, and transaction types.

**Warning signs:**
- Financial totals that don't match Airbnb dashboard figures
- Dates parsing to invalid values or year 1900
- Numeric fields reading as strings after import
- More or fewer columns than expected in a header row

**Phase to address:**
Platform data ingestion phase — build schema validation and raw file archiving from the first implementation.

### Pitfall 3: Airbnb Payout Timing Makes Reconciliation Non-Trivial

**What goes wrong:**
Airbnb does not pay hosts in one lump sum per booking at a predictable time. As of 2025: standard payout is ~24 hours after guest check-in for stays under 28 nights; monthly for stays 28+ nights; delayed until after checkout for new hosts (fewer than 2 completed stays); and potentially held indefinitely under expanded force majeure or fraud provisions. Additionally, Airbnb introduced Buy Now Pay Later (BNPL) options for guests, meaning a confirmed booking may cancel if a guest misses a payment installment — the cancellation arrives as a separate transaction event, not a modification to the original booking record. Starting October 2025, hosts using a PMS are automatically moved to host-only fee (15.5%) instead of the split fee (~3% host / ~14% guest). This changes the gross-to-net math if your parser was calibrated on the old split-fee model.

**Why it happens:**
Developers model one booking = one payout. Airbnb's actual payout model is event-driven with multiple asynchronous transactions: booking creation, payout release, potential adjustment, potential reversal. These arrive as separate line items in the transaction CSV with different dates and reference numbers that must be linked by confirmation code.

**How to avoid:**
- Group all transaction rows by confirmation code before calculating net income per booking.
- Never treat an unmatched payout as a new booking — match payouts to existing booking records.
- Track booking status separately from financial status: a cancelled booking may still have an associated payout that arrived before the cancellation.
- Build an "unreconciled transactions" queue for amounts that don't match a known booking.
- Flag the fee model (split vs. host-only) as a configuration field; document when it changed for your account.

**Warning signs:**
- Payout in Mercury that has no matching booking record
- A booking appearing with zero or negative net income
- Duplicate entries when Airbnb issues an adjustment
- P&L totals diverging from Mercury bank balance

**Phase to address:**
Accounting and reconciliation phase — treat booking-to-payout matching as a first-class feature, not an afterthought.

### Pitfall 4: VRBO Messaging Requires API Access That Is Not Publicly Available

**What goes wrong:**
The project requires sending automated welcome and arrival messages through platform messaging systems. VRBO does not have a public messaging API available to individual hosts. There is no VRBO-endorsed API connection for public use for messaging. Automated messages are also NOT sent for reservations synced via iCal — the iCal connection only syncs calendar availability, not messaging triggers. To send messages programmatically, you need API access that VRBO currently only grants to approved channel managers and PMS vendors.

**Why it happens:**
VRBO's help documentation describes message templates and "scheduled messages" as a host feature — but this refers to their in-app scheduler, not an API endpoint. Developers confuse the in-app UI capability with programmatic API access.

**How to avoid:**
For VRBO: messaging automation via direct API is not achievable for non-approved partners. Practical alternatives:
1. Use VRBO's built-in scheduled message templates (configured once, not dynamic per booking) for standard messages.
2. Trigger a human-review notification (email/push) when a VRBO booking arrives, with the pre-composed message text ready to copy-paste.
3. Accept that VRBO messaging will be semi-automated (template + manual send) in v1.

For Airbnb: the messaging API is available to approved partners only. However, Airbnb's built-in Automated Messages feature (accessible from the hosting dashboard) can schedule messages by booking event trigger. This is a UI configuration, not an API call — but it achieves the same result for standard messages.

Scope the "platform messaging" feature to explicitly exclude VRBO API messaging in v1, documenting it as a known limitation.

**Warning signs:**
- Attempting to call VRBO messaging endpoints and receiving 401/403 errors
- VRBO listing guest names not appearing in iCal data (they aren't — only dates are exported)
- Documentation describing VRBO messaging that only applies to approved PMS vendors

**Phase to address:**
Platform messaging phase — design around what is actually achievable before implementation; define v1 scope explicitly.

### Pitfall 5: LLM Hallucination on Financial Numbers

**What goes wrong:**
Local Ollama models confidently produce incorrect numbers when asked financial questions. LLMs do not perform arithmetic — they predict token sequences based on training patterns. "What is my net income for Q4?" may return a plausible-sounding but incorrect number that is a blend of training patterns and partial context. This is especially dangerous when Kim (the non-technical user) treats the answer as authoritative without cross-checking.

Specific failure modes:
- The LLM sums numbers from context but gets the arithmetic wrong on large sums
- The LLM answers with a number from a different time period if the query is ambiguous
- The LLM "remembers" a number from earlier in the conversation and uses it incorrectly later
- The LLM does not surface uncertainty — it gives a confident wrong answer rather than saying "I don't know"

**Why it happens:**
LLMs are pattern completion engines. They are not calculators. When given financial data in context, they attempt to complete the pattern of "a question about money + numbers in context = a reasonable-sounding monetary answer." The model's confidence calibration does not reflect actual arithmetic accuracy.

**How to avoid:**
- Never let the LLM do arithmetic. The LLM interface should translate natural language queries into structured queries (SQL or filtered lookups against the database), run the query, and present the verified numerical result as context for the LLM to describe in natural language.
- Architecture pattern: LLM → query intent detection → SQL query execution → inject result → LLM → natural language response. The LLM never sees raw numbers and is asked to add them; it sees a pre-computed result and is asked to describe it.
- Always show the source data that produced the answer (e.g., "Based on 12 Airbnb payouts in Q4 totalling $X...") so Kim can sanity-check.
- Add a disclaimer to every LLM response: "Verify important figures in the dashboard."

**Warning signs:**
- LLM giving different totals for the same query asked twice
- LLM answering questions about Q3 when asked about Q4 (temporal confusion)
- Numbers in LLM response not matching dashboard totals

**Phase to address:**
LLM query interface phase — establish the "LLM never does arithmetic" rule as a hard architectural constraint before writing a single line of the query interface.

### Pitfall 6: PDF Form Fields Filled But Not Rendered (NeedAppearances Trap)

**What goes wrong:**
pypdf (and related libraries) write values into PDF form fields, but opening the resulting PDF shows blank or partially blank form fields until the user clicks on them in a PDF viewer. This happens because the form appearance stream is not regenerated by the library — the field has a value but no appearance dictionary update. The PDF viewer is supposed to regenerate the appearance when NeedAppearances is True, but many email clients and PDF renderers (especially the ones resort staff likely use) do not re-render — they display blanks.

The consequence is that the resort Welcome Center receives a PDF that looks empty and the booking submission is treated as incomplete.

**Why it happens:**
The pypdf documentation recommends `auto_regenerate=False` but notes it defaults to `True` for legacy compatibility. Setting it to `True` correctly sets the NeedAppearances flag, which tells compliant viewers to re-render — but non-compliant viewers don't. The field contains data but the visual layer is missing.

**How to avoid:**
- Use PyMuPDF (fitz) instead of pypdf for form filling: PyMuPDF generates new appearance streams when it fills fields, producing PDFs that display correctly in all viewers without depending on the NeedAppearances flag.
- After filling, flatten the PDF (convert form fields to static content) before sending as email attachment. A flattened PDF has no form fields — just rendered text — and displays identically everywhere.
- Test the filled PDF by opening it in macOS Preview, Adobe Reader, and a mobile email client before relying on the pipeline. These three cover the range of rendering behaviors.
- Verify the resort's form is AcroForm-based (not XFA/LiveCycle). XFA forms require different libraries (or conversion to AcroForm first).

**Warning signs:**
- PDF opens in macOS Preview looking blank, but clicking fields shows they have values
- Resort staff report receiving blank booking forms
- PDF looks correct in Adobe Reader but blank in iOS Mail attachment preview

**Phase to address:**
PDF automation phase — validate the filled PDF rendering in all target viewers before sending a single submission to the resort.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Hardcode platform-specific column names | Fast to implement | Parser breaks silently on format change; requires code deploy to fix | Never — use config-driven column mapping from day one |
| Store secrets in `.env` committed to git | Easy setup documentation | Credentials exposed if repo ever goes public; project is intended for GitHub | Never — use `.env.example` + `.gitignore` + documented setup steps |
| Skip raw CSV archiving | Simpler storage | Cannot re-parse historical data when parser changes; cannot audit discrepancies | Never — store raw files, they are small |
| Use iCal as the booking data source | Single URL to poll | Missing all guest details; cannot drive PDF or messaging automations | Never for this use case |
| Let LLM compute totals inline | Simpler query pipeline | Hallucinated financial figures delivered to non-technical user with no recourse | Never — always SQL-compute, LLM-describe |
| Anonymous Docker volumes | Zero config | Impossible to identify, backup, or migrate; data lost on container recreation | Never for persistent data |
| Single Docker volume for all data | Simpler compose file | Cannot apply different backup schedules to DB vs. logs vs. raw files | Acceptable for MVP if backup covers everything; migrate to separate volumes in v2 |
| Hard-coded property names/addresses | Faster first pass | Violates open-source goal; must touch code to reconfigure | Never — externalize all property data to config on day one |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Airbnb transaction CSV | Trusting column order; parsing without schema validation | Validate header row on every ingest; alert on unexpected schema |
| Airbnb iCal | Using iCal as guest data source | Use iCal only for blocked-date awareness; use email parsing or earnings CSV for booking details |
| Airbnb payouts | Treating one booking = one payout row | Group by confirmation code; handle adjustments, reversals, and holds as separate linked events |
| VRBO messaging | Calling a VRBO messaging API that doesn't exist for non-partners | Use VRBO's in-app scheduled templates for standard messages; notify host for non-standard sends |
| Mercury bank | CSV export format assuming stable column names | Schema-validate Mercury CSV headers; Mercury supports a proper API (`mercury.com/api`) — prefer API over CSV if authentication is feasible |
| Mercury bank API | Assuming no rate limits exist | Check `docs.mercury.com` for current rate limits before building polling logic |
| RVshare | Assuming API access similar to Airbnb/VRBO | RVshare has no known public host-facing booking API; budget for manual CSV download or email parsing as fallback |
| Email sending (resort PDF) | Using residential ISP SMTP relay | Residential IP blocks are common; use authenticated SMTP relay (Gmail SMTP, SendGrid free tier, or similar) with SPF/DKIM configured |
| PDF form filling | Testing only in Adobe Reader | Test in macOS Preview, iOS Mail attachment view, and Adobe Reader — rendering differs significantly |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Polling all platform exports on every cron run | Slow startup, large memory use, redundant re-parsing | Track last-processed timestamp; only process new records on incremental runs | With ~50+ bookings history the latency becomes noticeable; with hundreds it becomes unacceptable |
| Loading entire transaction history into LLM context | Token limit exceeded; slow responses | Query the database to retrieve only relevant rows; pass computed results to LLM, not raw data dumps | Ollama context window filled after ~6 months of transaction history depending on model |
| Sending PDF via email on every cron cycle | Duplicate submissions to resort | Track submission status per booking in DB; only submit once; mark as submitted with timestamp | Immediately — resort will receive duplicate forms for the same guest |
| Re-running Mercury reconciliation without idempotency | Duplicate ledger entries | Assign a stable unique ID to each transaction on first ingest; use upsert not insert | First time a re-run occurs (crash recovery, re-import) |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Committing `.env` file with real credentials to git | All platform API keys, SMTP passwords, and bank API tokens exposed publicly | Add `.env` to `.gitignore` on day one; provide `.env.example` with placeholder values only |
| Exposing Docker web port (80/443) to the internet | Dashboard accessible from internet; brute-force possible | Bind only to `127.0.0.1` or local subnet; access via VPN or local network only |
| Using `docker inspect` revealing secrets via env vars | Secrets visible to anyone with Docker daemon access | Use Docker Secrets or volume-mounted secret files instead of env vars for sensitive values; limit Docker socket access |
| Storing raw Mercury/Airbnb CSV files in web-accessible directory | Guest PII and financial data accessible via URL | Store raw files outside the web root or in a volume not mounted to the web server |
| Storing guest personal data (email, phone) without considering data retention | Privacy risk; PII in perpetuity | Define a retention policy (e.g., delete guest contact details 12 months after stay); document it clearly in the config |
| Hardcoding property details (lock codes, addresses) in code | Lock codes exposed in public git repository | All property-specific data in config files; config files excluded from git; `.env.example` shows structure without real values |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| LLM response with no source attribution | Kim cannot verify if a financial figure is correct; trust erodes after first wrong answer | Every LLM response shows the underlying data rows that produced the answer |
| LLM query accepting vague time ranges | "This year" means different things in January vs December; wrong period returned | Always resolve relative dates to absolute date ranges before querying; show the resolved range in the response |
| Dashboard showing totals that don't match platform figures | Confuses Kim; undermines confidence in the entire system | Show reconciliation status indicator; highlight unmatched transactions clearly |
| Automation silently fails (PDF not sent, message not sent) | Resort deadline missed; guest not welcomed | All automation outcomes logged with success/failure status; dashboard shows pending actions needing human attention |
| Error messages in developer-friendly language | Kim sees stack traces or JSON errors | User-facing error messages are human-readable; log technical details separately |

## "Looks Done But Isn't" Checklist

- [ ] **Airbnb ingestion:** Does it correctly handle adjustments, resolutions, and cancelled booking transactions — not just successful payouts?
- [ ] **Reconciliation:** Are all Mercury transactions matched to a booking or explicitly categorized as "unrelated expense"? Is there an "unmatched" bucket visible in the dashboard?
- [ ] **PDF form:** Does the filled PDF display correctly when opened as an email attachment on iOS (not just in Adobe Reader on desktop)?
- [ ] **PDF submission:** Is duplicate submission prevented? Does the system mark bookings as "submitted" and skip them on subsequent runs?
- [ ] **Messaging:** For VRBO and RVshare bookings, is there a clear indication in the dashboard that messaging is manual/semi-automated so Kim knows to act?
- [ ] **LLM queries:** Do all financial figures in LLM responses trace back to a verifiable database query? Is it impossible for the LLM to output a number it computed itself?
- [ ] **Docker data persistence:** Are all stateful directories mapped to named volumes? Does a `docker compose down && docker compose up` retain all data?
- [ ] **Backup:** Is there a tested restore procedure? Has a full restore from backup actually been verified to work?
- [ ] **Config open-source safety:** Does `git log -- config/` contain any real property data, lock codes, or credentials? Is `.gitignore` correct from the first commit?
- [ ] **RVshare coverage:** Is there a documented fallback for RVshare bookings if no export mechanism works? Can bookings be entered manually without breaking the rest of the pipeline?

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Parser broken by CSV format change | MEDIUM | Restore from raw file archive; update parser; re-ingest; validate totals |
| Duplicate transaction entries from re-import | LOW | Add unique constraint on transaction ID; run deduplication query; correct ledger totals |
| PDF submitted twice to resort | LOW | Email resort Welcome Center to disregard the duplicate; update submission tracking |
| LLM gave Kim a wrong financial figure | MEDIUM | Show corrected figure from dashboard; explain the discrepancy; add safeguard to prevent recurrence |
| Docker volume lost (no named volume / accidental `docker compose down -v`) | HIGH | Restore from backup; if no backup, rebuild from raw CSV archives (if preserved); re-ingest all transactions |
| VRBO messaging not sent (missed arrival window) | MEDIUM | Send message manually through VRBO inbox; update arrival message trigger timing |
| iCal-only approach discovered to be insufficient mid-build | HIGH | Pivot to email parsing for booking details; adds 1-2 week delay; all downstream code assuming guest fields in iCal must be reworked |
